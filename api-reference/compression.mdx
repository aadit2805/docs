---
title: 'Gzip compression'
description: 'Reduce response size by compressing large JSON payloads with gzip'
---

The API includes an optional gzip compression middleware for JSON responses. When enabled, it reduces bandwidth for large payloads while leaving small responses untouched.

## How it works

The middleware compresses a response only when **both** conditions are met:

- The client sends an `Accept-Encoding` header that includes `gzip`.
- The serialized JSON payload is at least **1 KB** (1024 bytes).

When the response is compressed, the middleware sets:

- `Content-Encoding: gzip`
- `Content-Type: application/json`

Responses that do not meet both conditions are sent uncompressed, unchanged.

## Requesting a compressed response

Send the `Accept-Encoding: gzip` request header:

```bash
curl -H "Accept-Encoding: gzip" https://api.example.com/resource --compressed
```

Most HTTP clients (browsers, `fetch`, `axios`, `requests`) request and decode gzip automatically.

## Example

```http Request
GET /resource HTTP/1.1
Host: api.example.com
Accept-Encoding: gzip
```

```http Response
HTTP/1.1 200 OK
Content-Encoding: gzip
Content-Type: application/json

<gzip-compressed body>
```

## Notes

- Payloads smaller than 1 KB are returned uncompressed to avoid overhead.
- Clients that omit `Accept-Encoding: gzip` always receive an uncompressed JSON response.
