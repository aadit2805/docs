Sift uses **Supabase** (Postgres \+ `pgvector` reserved for future RAG) as its single source of truth. Migrations live in `supabase/migrations/`.

## Core tables

| Table | Purpose |
| --- | --- |
| `courses` | Course catalog (code, title, credits, `prereqs` JSON) |
| `sections` | Per-semester course offerings (sparse in V1) |
| `professors` | Faculty with RateMyProfessors metadata |
| `grade_distributions` | Per-(course, professor, semester) GPA distribution |
| `degree_plans` | Per-major degree requirement structure |
| `users` | Clerk-linked user profile (completed courses, preferences) |
| `semester_plans` | V2 multi-semester plans saved as JSONB |

## Migrations

| File | Purpose |
| --- | --- |
| `001_initial_schema.sql` | Core tables: courses, sections, professors, grade\_distributions, degree\_plans |
| `002_clerk_auth.sql` | `users` table keyed by Clerk user ID |
| `003_semester_plans.sql` | Multi-semester plan storage |

When changing the schema:

1. Add a new numbered migration file
2. Apply it in Supabase
3. Sync the type changes in **both** `client/src/lib/types.ts` and `server/src/types/index.ts`

## The scraper

`server/scripts/scrape.ts` populates the database from two sources:

- **Anex** — grade distributions per course/professor/semester
- **RateMyProfessors** — professor ratings, would-take-again, difficulty

Run it with:

```bash
make scrape
```

The scraper is idempotent — re-running upserts existing rows rather than duplicating.

## Conventions

- **Grade distributions** stored as decimals (0–1); the UI multiplies by 100 to render percentages.
- **Course codes** are stored as `DEPT NNN` (e.g. `CSCE 121`) — keep this format consistent in seed data.
- **Prereqs** are stored as a JSON expression on the course row. There's no separate prereq table — the DAG is built in-memory by `prereq-graph.ts`.
