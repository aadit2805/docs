The V2 planner generates **valid multi-semester degree plans** from a student's completed courses \+ remaining requirements. Plans are saved as JSONB to the `semester_plans` table and scoped per-user.

## End-to-end pipeline

<Steps>
  <Step title="Build the prerequisite DAG">
    `prereq-graph.ts` reads `courses.prereqs` and constructs an in-memory DAG. There's no separate prereq table — keeping the source of truth on the course row simplified the scraper and avoids drift.
  </Step>
  <Step title="Run the greedy solver">
    `solver.ts` walks future semesters one at a time. For each, it:

    1. Computes the **ready set** — courses whose prereqs are satisfied and that are offered in that season.
    2. Sorts the ready set by the active **priority function**.
    3. Assigns the top-N up to the **credit limit**.
    4. Adds those to the "completed" set for the next iteration.
  </Step>
  <Step title="Repeat with four objectives">
    The solver runs **four times** with different priority functions:

    - `fastest` — finish as early as possible (uses critical-path distance)
    - `highest_gpa` — maximize expected course GPA
    - `balanced` — even credit load across semesters
    - `best_professors` — prioritize professor rating
  </Step>
  <Step title="Score and return">
    Each plan is scored against the same weighted formula used by recommendations. The four plans \+ scores are returned to the client; the user picks one and saves it.
  </Step>
</Steps>

## Storage

- **Table** — `semester_plans` (see migration `003_semester_plans.sql`)
- **Shape** — JSONB containing `{ semesters, scores }`
- **Scoping** — every row is keyed by `user_id`; deletes cascade

## API surface

| Method | Path | Purpose |
| --- | --- | --- |
| `POST` | `/api/planner/generate` | Returns 4 plan variants |
| `POST` | `/api/planner/save` | Persists a chosen plan to `semester_plans` |
| `GET` | `/api/planner/plans` | Lists the user's saved plans |
| `DELETE` | `/api/planner/plans/:id` | Removes a saved plan |

All planner endpoints require auth.

## Known limits

- The solver is **greedy**, not optimal. It's fast and good-enough; a constraint solver would be marginally better in pathological cases.
- Killer-combo detection runs **after** plan generation. The solver doesn't avoid them proactively — re-run with a different objective if you don't like the warning.
- **No section data** — scheduling preferences (e.g. avoid 8 AMs) are disabled until we ingest the section catalog.
